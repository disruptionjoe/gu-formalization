## Purpose

This packet ranks the potential insight/test set from `32-potential-insights-novelty-and-tests.md` using five different evaluator personas.

The goal is not consensus theater. The goal is to decide:

- which tests agents should run now;
- which insights should be pushed toward collaborators when the GitHub repo is published;
- which ideas are too broad, too expensive, or likely dead ends;
- which heterodox lines are actually worth protecting because they may escape the usual gradient of already-found work.

## The Five Evaluator Personas

| Persona | What it values | What it penalizes |
|---|---|---|
| P1 Rigor Gatekeeper | precise definitions, theorem-class fidelity, falsifiability, clean negative results | metaphor, hidden strengthening, unfalsifiable grandeur |
| P2 Heterodox Dialectician | real Aufhebung, hidden-assumption reversal, unexplored axis drops, profound reframing | safe summaries, already-known literature reviews, shallow novelty |
| P3 Agent Lab Operator | tests that can be executed locally by agents in small packets | broad programs, expert-only proof work, dependency on external access |
| P4 Open-Source Field Builder | repo-worthy prompts that invite specialists to contribute | private-only artifacts, claims that sound cranky, work without onboarding affordances |
| P5 Strategic Research PM | value of the next 5-10 hours of work, sequencing leverage, decision clarity | sprawling work, low expected signal, work that cannot change the next decision |

Each persona scores 0-5:

- 0 = do not pursue
- 1 = park unless external reason appears
- 2 = useful background only
- 3 = worthwhile support work
- 4 = strong priority
- 5 = highest leverage

Final route:

- **Agent now** = bounded local work should happen before repo publication.
- **Repo push** = publish as a core open problem / collaborator invitation.
- **Repo appendix** = useful background, not a lead claim.
- **Park** = hold until a stronger source, collaborator, or decision appears.
- **Avoid** = likely dead end or reputation risk.

## What To Push In The GitHub Repo

Core public claims to emphasize:

1. **No-go theorems are class-relative.** The repo maps assumption classes and successful exits rather than claiming the theorems are wrong.
2. **The strongest synthesis is a substrate-invariant / forgetful-shadow program.** This should be presented as a research hypothesis, not as established literature.
3. **The Type II_1 spectral Standard Model bridge is genuinely missing.** The repo can invite operator algebra, NCG, and mathematical physics collaborators to fill or refute that bridge.
4. **The distributed-systems analogy is unexplored.** The repo should invite formalization through a pilot, probably Nielsen-Ninomiya first.
5. **Every proposal must specify its six axes.** This prevents vague "new physics" talk.
