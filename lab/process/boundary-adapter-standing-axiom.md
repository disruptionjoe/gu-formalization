# Boundary Adapter Standing Axiom

Joe's standing directive, verbatim in intent: we KNOW the construction needs
the boundary adapter. That is settled and no longer news. Work proceeds by
ASSUMING the adapter as an axiom and figuring out what must come through it
for all four legs to fit.

## The axiom

The boundary adapter exists as a typed interface (axiom `A_boundary`). All
construction-space, interior, and recovery work proceeds under this
assumption by default.

## The binding rules

1. **"Needs the boundary adapter" is never a stop, gate, park, or
   no-worthy-work reason.** A run that finds a leg requires the adapter
   records WHAT the leg requires the adapter to transmit (an entry in the
   adapter demand ledger) and CONTINUES the conditional construction. Any
   run output resting on "blocked: adapter needed" is a protocol violation,
   not a finding — we know that already.
2. Conditional grades (`R0_COND` / `R1_COND`) are the normal working grades,
   not a degraded mode. The lane's live question is: **does a possible
   construction exist where all four legs fit, given the adapter — and what
   exactly must the adapter deliver?**
3. GATED_P2C states gate only the final *unconditional* validation (which
   still requires a concrete instance with provenance). They do not gate
   conditional construction, hypothesis testing, or interior exploration.
4. The adapter demand ledger lives with the source-object interface
   contract; every new demand is typed, counted, and routed to
   possibility-to-capability as the sharpened build target.
5. Honesty boundary unchanged: conditional results are labeled conditional;
   no unconditional claim, canon verdict, or public posture moves until an
   instance validates the axiom. Assuming the adapter is a work posture,
   not a truth claim.
