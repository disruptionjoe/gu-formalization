# Recovery No-Go Defense Protocol

This protocol prevents a branch-local obstruction from being mistaken for an exhaustive no-go. It also
prevents unlimited reframing from protecting a failed construction. Every new recovery `NO_GO` first receives
a history audit and then a bounded defense sequence before the daily steward may call its predeclared
construction class exhausted.

The source methods are observations from the sibling `ai-epistemology` repository. They guide search but are
not evidence for GU. The load-bearing rules are:

- identify the construction fork and state which side produced the obstruction;
- give every kill a separate defense pass;
- vary assumptions and construction classes, not merely computational techniques;
- predeclare the plausible construction space and stop when it is exhausted;
- require a legitimate reframe to explain more, not merely avoid the failed test;
- preserve the original negative result unless a new construction passes its own frozen benchmark;
- follow any positive escape with an adversarial twin.

## Mandatory history audit before the three swings

Before banking, closing, or attempting to rescue a new no-go, search the prior GU record for the same
substantive obstruction. This audit does not count as one of the three broad swings.

1. Search canon, claim and hypothesis ledgers, corrections, explorations, tests, run receipts, and superseded
   steering history. Search by the mathematical or physical mechanism, not only by the current label.
2. Record every earlier encounter, its construction fork, assumptions, grade, outcome, and any mechanism that
   was said to clear or relocate it.
3. Check whether the earlier clearance still applies to the exact frozen construction now under test.
4. If the same obstruction was already cleared for the same construction, classify the new result as an
   `INTEGRITY_CONFLICT`. Reconcile the proofs and premises before treating either result as authoritative.
5. If it was cleared only for another construction, preserve both results and make the construction-relative
   boundary explicit.
6. If no material prior encounter exists, record `NO_PRIOR_CLEARANCE_FOUND` with the search paths and terms.

An absence-of-evidence statement is only as good as the declared search surface. Do not claim exhaustive repo
history from a narrow keyword search.

If the audit closes without an integrity conflict and enough run capacity remains, continue directly into Swing
1 in the same hourly run. The history audit still does not count as a defense swing; the typed scope-and-fork
work does. Do not create history-only receipts merely to increase the run count.

## Six-axis consideration for class-relative scope

Use the current form of GU's historically named six-axis method: Layer 0 semantic alignment plus L1 through L7
(substrate, observer, pairing, causal order, emergence, coordination loop, and positivity or state-space metric).

1. Give the failed construction a Layer-0 and L1-L7 signature. This states the exact class in which the no-go
   was obtained.
2. During the history audit, compare earlier clearances by signature. A prior result with a material axis change
   is a different construction, not a same-construction clearance.
3. For every proposed escape, state which axes change, which stay fixed, and which assumption of the no-go the
   change is intended to break. A name change with no axis delta is not a new construction.
4. Before Swing 2 admits an escape candidate, complete the current template, including literature anchors, the
   bridge claim appropriate to the target, and one first falsification test.
5. Blank or vague axes return `UNDERDEFINED`; they do not count as either a rescue or evidence that the no-go is
   class-invariant.

The six-axis map is a specification and scope discipline, not positive evidence. Its existing escape-hatch map
also contains class-invariant hard cores, so an axis change must face the original benchmark and adversarial twin.

## Minimum three-swing sequence

After the history audit, each branch-local no-go receives at least three substantial hourly swings. A run may
do more than one linked check, but it may not count prose, queue maintenance, persona voting, the history audit,
or a relabeling as a swing.

### Swing 1: type, minimize, and fork the obstruction

1. Consume the completed history audit and resolve any `INTEGRITY_CONFLICT` before proceeding.
2. Reproduce the no-go from its smallest load-bearing assumptions.
3. Name the exact object, equation, representation, variation space, boundary conditions, and observable used.
4. Declare the construction space before attempting a rescue. At minimum compare the applicable GU-native
   construction, the standard-physics comparator, a boundary-conditioned construction, and any already-named
   GU action or quotient fork. Mark inapplicable members explicitly.
5. Record the Layer-0 and L1-L7 signature of the failed construction and the proposed axis delta for every
   alternative.
6. Identify the one assumption or channel whose change would overturn the result.

Allowed outcomes: `SCOPE_CONFIRMED`, `CLASS_RELATIVE`, `UNDERDEFINED`, or `INVALID_NO_GO`.

### Swing 2: attempt a genuinely different construction

Choose the best matched escape moves from the AI-epistemology registry. Candidates include Type-the-Residue,
Strong-to-Bounded No-Go, Test-the-Close, Analytic Reduction, Target Relocation, Foundation or Model Swap,
Operation Retyping, reopenable walls under a new lens, and reverse-from-fixed-point when its form-versus-
structure guard applies.

The attempt must change a load-bearing construction or derive a previously absent object. Changing notation,
fitting a coefficient, importing the target theory, or adding the desired verdict as boundary data does not
count. An admitted escape must complete the current Layer-0 plus L1-L7 template and a first falsification test.
A survivor must explain at least one fact the failed construction did not explain.

Allowed outcomes: `NO_SURVIVOR`, `SURVIVOR_CANDIDATE`, `NEW_DEPENDENCY`, or `INVALID_ESCAPE`.

### Swing 3: adversarial adjudication and stop decision

1. Run an adversarial twin against every survivor using the same frozen benchmark that killed the original.
2. Check for sector-specific retuning, hidden imports, presentation equivalence, and incompatible branch mixing.
3. Compare explanatory reach, not just logical survival.
4. Record an exact resurrection trigger if the predeclared construction space is exhausted.

Allowed outcomes: `CLASS_EXHAUSTED`, `BOUNDED_NO_GO`, `REFRAMED_SURVIVOR`, or `MORE_CONSTRUCTION_SPACE`.

`MORE_CONSTRUCTION_SPACE` is lawful only when the run identifies a new source-owned construction and a finite
test. Repeating the same obstruction with a new name is not progress.

## Interaction with claim status

The original result remains valid at its stated branch and grade throughout the defense sequence. A successful
escape creates a new construction candidate; it does not erase the old counterexample. A failed defense may
strengthen the scope only through the ordinary GU claim-status workflow. This protocol alone changes no canon,
verdict, scientific status, `bar(b)`, or H59 state.

## Conditional physical unitarity question

The boundary adapter has two separable burdens:

1. p2c must independently construct and return a valid adapter for an unconditional physical claim.
2. GU must test conditional sufficiency: given a precisely typed adapter interface and favorable branch inputs,
   does one frozen GU construction actually produce the physical field complex, constraint or BRST quotient,
   positive state space, observables, probabilities, locality, and unitary or state-preserving dynamics?

GU owns the second question now. A conditional success is informative and remains explicitly assumption-capped.
It must list every additional datum needed beyond the adapter. If further sector-specific inputs are required,
then the adapter is not sufficient for that frozen construction.
